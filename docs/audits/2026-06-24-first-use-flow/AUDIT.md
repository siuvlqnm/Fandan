# 饭单首次使用与核心流程审计

审计日期：2026-06-24  
审计模式：移动端 UX + 可访问性风险联合审计  
审计环境：本地 production build，Codex 内置浏览器，目标视口 `390 × 844`，全新测试账号与本轮新建数据  
用户目标：第一次打开饭单后，尽快完成“一顿饭吃什么”并获得可用购物清单

## 结论

当前产品已经具备完整的数据能力和一致的视觉语言，但首次使用仍以内部实体为中心：工作区、对象、菜品、饭单、邀请、确认、清单同时暴露。用户必须先理解产品结构，才能完成一顿饭。

最严重的问题不是视觉装饰，而是：

1. 首次价值路径过长，要求用户连续理解多个概念。
2. 空页面仍优先展示搜索和筛选，辅助工具压过主体内容。
3. 表单口头承诺“只填名称即可”，实际仍把所有可选字段放在提交按钮前。
4. 底部导航直接暴露“菜品”和“对象”两个内部数据模型。
5. 菜品缺少“这份食材对应几人份”的基准，购物清单直接按饭单份数倍增，存在采购量错误风险。

建议暂停继续增加 Phase 10 功能，先重构首次使用和核心任务流。

## 已确认的优点

- 整体颜色、圆角、排版和图标语言一致，移动端视觉基线已经形成。
- 主要按钮通常足够醒目，底部导航位置稳定。
- 空状态有解释和下一步，不会出现完全空白页面。
- 邀请页和购物清单页的单一任务感明显更好，可以作为后续精简的参照。
- 页面在本轮目标视口没有横向溢出。

## UX 风险

### P0：首次价值路径不是产品主线

当前路径接近：

> 注册 → 工作区 → 对象 → 菜品 → 饭单 → 分享 → 确认 → 购物清单

用户真正想完成的是：

> 安排一顿饭 → 选几个菜 → 知道要买什么

工作区、对象、菜品档案和分享权限应当成为系统自动处理或按需出现的能力，而不是前置知识。

### P0：购物清单数量缺少可信计算基础

菜品食材只有数量和单位，没有“该菜谱默认对应几人份”。饭单项目又有份数，生成器会直接执行食材数量乘份数。本轮录入 `番茄 3 个、鸡蛋 4 个`，饭单选择 `3 份` 后，清单显示 `番茄 9 个、鸡蛋 12 个`。

用户无法判断菜品食材是单人量还是整道菜的量。这不仅增加学习成本，还可能直接导致采购错误。必须先确定产品语义：

- 菜品记录基准份数，再按比例换算；或
- 首版不自动倍增，只原样汇总并明确提示；或
- 创建饭单时让用户确认实际采购量。

### P0：注册后没有真正的首次成功路径

空工作台提供多个入口，但没有把用户带完一次任务。首用应当是一个连续流程，而不是把用户丢到功能导航中自行组合。

### P1：空状态仍以管理工具为主

对象、菜品、饭单列表在数量为零时仍显示完整筛选器。筛选器只有在数据足够多时才有价值。零数据和少量数据时应直接显示内容或创建引导。

### P1：主导航暴露数据结构

五个一级入口中，“菜品”和“对象”是被饭单使用的资料，不一定是普通家庭每天主动管理的任务。它们占据与工作台、饭单同等权重，会强化“我需要先准备资料”的误解。

### P1：同一页面存在过多并列操作

- 工作台同时提供新建饭单、对象、菜品、饭单列表、邀请和购物清单。
- 饭单详情同时提供统计、反馈、清单、四个页签、加菜、排序、菜品详情和删除。
- “我的”同时承担账号、工作区、成员、邀请和权限说明。

用户需要不断判断“现在应该点哪个”，而不是顺着任务自然前进。

### P1：专业服务场景压过普通家庭

对象表单里的客户、预算、联系安排，以及新建饭单的对象选择，明显服务于私厨/做饭阿姨。对普通家庭，它们应隐藏在“用餐档案”或专业模式中。

### P2：解释文字多，但没有减少决策

很多页面通过段落解释复杂模型，却仍保留所有字段和入口。说明文字进一步占用首屏，并不能替代真正的渐进式展示。

## 可访问性风险

以下是截图和只读 DOM 能确认或提示的风险，不代表完整 WCAG 审核：

- 饭单详情的“加菜”、上移、下移、菜品和移除操作实测高度为 `32px`，返回链接为 `36–40px`；低于移动端常用的约 `44px` 触控目标建议。
- 浅绿色背景上的灰色说明文字、浅色标签和部分边框存在对比度不足风险，需要用实际颜色值计算验证。
- 多个页面同时出现顶部返回、卡片内返回/操作和底部导航，键盘与读屏顺序可能冗长。
- 页签和状态变化在 DOM 中有名称，但焦点移动、错误播报、加载状态和动态更新是否被辅助技术感知，本轮截图无法确认。
- 表单虽然有可访问名称，但可选字段数量过多会增加认知负担；这属于可访问性与一般 UX 的共同问题。

## 建议的目标流程

### 首次使用

1. 独立注册页，只收名称、邮箱、密码。
2. 注册成功直接进入“安排第一顿饭”。
3. 默认对象就是当前家庭，不展示工作区和对象概念。
4. 只要求一个饭单名称或直接使用“今晚”，然后内联添加菜名。
5. 创建后立即展示饭单和可用购物清单。
6. 完成第一次价值后，再提示保存菜品详情或邀请家人。

目标：一分钟内，最多两个必填决策，完成第一顿饭。

### 常用导航

建议优先验证三项一级导航：

- 首页：下一顿饭和一个主要按钮。
- 饭单：当前与历史安排。
- 我的：账号与家庭。

菜品库、用餐档案、邀请和工作区管理从相关任务中进入，或放在“我的/更多”中，不再与核心任务同级。

### 页面密度规则

- 每页一个主要目标和一个主按钮。
- 零数据时不显示筛选器。
- 少量数据时使用一行搜索；高级筛选按需展开。
- 首屏至少露出核心内容或当前任务，而不是只露出工具。
- 可选字段放入“更多设置”，不得挡在提交按钮前。
- 普通家庭默认值自动完成；专业服务字段在需要时才出现。

## 建议实施顺序

1. 修正购物清单数量语义，避免真实采购错误。
2. 重构首次注册与“第一顿饭”连续流程。
3. 把工作区和对象从普通家庭主路径中隐藏。
4. 精简新建饭单，只保留当前任务必要字段。
5. 重构底部导航和页面职责。
6. 清理所有列表页的首屏搜索/筛选占用。
7. 最后统一压缩卡片、说明文字、间距和次要操作。

## 证据限制

- 本轮使用本地 production build 和本轮创建的数据，没有验证线上网络延迟、真实邮件、跨设备邀请或生产错误恢复。
- 没有执行完整键盘、屏幕阅读器、缩放和颜色对比度测试。
- 没有访谈真实家庭用户；关于学习成本的判断来自当前任务路径、界面证据和本轮使用体验。
