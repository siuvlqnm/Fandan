# 饭单 MVP 开发任务清单

## 1. 开发目标

实现饭单 MVP 的核心闭环：

> 登录 -> 创建用餐对象 -> 创建菜品 -> 创建饭单 -> 添加菜品 -> 生成购物清单 -> 分享确认 -> 查看反馈 -> 完成归档 -> 复制复用

## 2. 推荐技术栈

- 应用框架：SvelteKit + Svelte 5
- 样式：Tailwind CSS
- 组件：shadcn-svelte
- 认证与权限：Better Auth
- 表单：Superforms + Zod 或 Valibot
- 状态管理：Svelte 5 runes + SvelteKit load/actions，必要时引入 TanStack Svelte Query
- 后端：SvelteKit server routes/actions 优先
- 部署：Cloudflare Workers / Pages
- 数据库：Cloudflare D1
- ORM：Drizzle
- API：RESTful API

Hono 不作为 MVP 默认后端。后续需要开放独立 API、小程序接口、移动端接口或独立 worker 服务时，再引入 Hono。

## 7. 风险点

### 首次使用太重

风险：用户一开始就要录对象、菜品、食材，会放弃。

处理：允许从新建饭单开始，过程中顺手创建对象和菜品。

### 菜品库维护成本高

风险：用户不愿意认真录菜谱。

处理：第一版只要求菜名，食材和做法可选。

### 购物清单不够准确

风险：不做单位换算时，清单看起来不智能。

处理：定位为“可编辑初稿”，第一版重在减少整理成本。

### 服务者和家庭用户入口混乱

风险：一开始问身份会让用户困惑。

处理：统一入口为“新建饭单”，用用餐对象类型自然分流。

### 分享确认页过复杂

风险：客户不愿意操作。

处理：分享页只保留查看、反馈、确认，不要求注册。

## 8. 第一版成功标准

产品层面：

- 用户能在 3 分钟内创建并分享一份饭单
- 访客打开链接后 30 秒内能完成反馈或确认
- 购物清单能减少用户手动整理时间
- 历史饭单能被复制复用

验证层面：

- 至少 5 位真实用户完成试用
- 至少 3 位服务者愿意继续使用
- 至少 1 位服务者表达明确付费意愿
- 用户反馈中“比微信沟通清楚”成为高频评价
