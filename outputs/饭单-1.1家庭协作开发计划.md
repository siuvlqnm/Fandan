# 饭单 1.1 家庭协作开发计划

## 1. 版本目标

把当前仅用于数据隔离的默认 `space` 升级为真正可供一家人共同操作的家庭工作区，并补齐 1.0 上线后的关键产品能力。

核心闭环：

> 创建家庭工作区 -> 邀请家人 -> 成员加入 -> 共同维护菜品与饭单 -> 一起勾选购物清单 -> 能看出谁做了什么

Linear 项目：[饭单 1.1 家庭协作](https://linear.app/less-lab/project/饭单-11-家庭协作-e38e53b6ad2a)

Linear 文档：[饭单 1.1 家庭协作开发计划](https://linear.app/less-lab/document/饭单-11-家庭协作开发计划-9c46d4025c3a)

## 2. 身份边界

- 家庭成员：拥有账号，加入工作区后可长期操作共享数据。
- 用餐对象：被安排吃饭的家庭、客户或聚餐对象，不要求拥有账号。
- 分享访客：通过单份饭单链接临时查看、反馈或确认，不自动成为工作区成员。

第一版成员角色只保留两种：

- `owner`：管理工作区、成员、邀请和危险操作。
- `member`：共同维护对象、菜品、饭单和购物清单。

暂不引入复杂的 admin/editor/viewer 权限矩阵，等真实家庭协作反馈出现后再扩展。

## 3. 交付阶段

### Phase 9：家庭工作区基础

目标：家人可以通过邀请加入同一工作区，并安全访问同一份业务数据。

推荐执行顺序：

1. [LES-104](https://linear.app/less-lab/issue/LES-104) 建立成员、邀请与当前空间数据模型。
2. [LES-105](https://linear.app/less-lab/issue/LES-105) 重构认证上下文和角色校验。
3. [LES-100](https://linear.app/less-lab/issue/LES-100) 实现邀请、接受和加入流程。
4. [LES-101](https://linear.app/less-lab/issue/LES-101) 实现工作区设置和成员管理页面。
5. [LES-103](https://linear.app/less-lab/issue/LES-103) 支持创建和切换多个工作区。
6. [LES-102](https://linear.app/less-lab/issue/LES-102) 完成双账号端到端验证和旧数据迁移保护。

阶段验收：

- 现有 1.0 用户升级后仍能看到原数据。
- owner 可以生成邀请，已有用户或新用户都能加入。
- owner 和 member 可以共同维护同一工作区的数据。
- 非成员不能通过猜测 ID 跨空间访问。
- “我的”导航进入真实的工作区与成员设置页面。

### Phase 10：家庭协作体验

目标：共同操作可辨认、可追踪，并避免静默覆盖。

- [LES-106](https://linear.app/less-lab/issue/LES-106) 记录并展示关键操作归属。
- [LES-107](https://linear.app/less-lab/issue/LES-107) 增强共同采购体验并显示勾选成员。
- [LES-108](https://linear.app/less-lab/issue/LES-108) 增加多人修改版本冲突检测。
- [LES-109](https://linear.app/less-lab/issue/LES-109) 增加家庭动态与待处理事项。

阶段验收：

- 成员可以看出菜品、饭单和购物项最近由谁处理。
- 两位成员同时编辑时不会静默覆盖对方修改。
- 工作台只展示真实的工作区动态和待办。

### Phase 11：1.1 产品补全

目标：补齐正式使用所需的账号、分享、清单入口和上线保障。

- [LES-110](https://linear.app/less-lab/issue/LES-110) 在分享面板提供权限和有效期设置。
- [LES-111](https://linear.app/less-lab/issue/LES-111) 补齐账号设置、邮箱验证与密码恢复。
- [LES-112](https://linear.app/less-lab/issue/LES-112) 建立购物清单中心和历史入口。
- [LES-114](https://linear.app/less-lab/issue/LES-114) 完善可观测性、自定义域名和数据保护。
- [LES-113](https://linear.app/less-lab/issue/LES-113) 建立自动化浏览器回归和发布门禁。

阶段验收：

- 用户可以恢复账号并区分个人设置和家庭设置。
- 分享链接权限与到期时间可以在页面配置。
- 用户不用先找到饭单，也能进入当前购物清单。
- 家庭协作完整链路有稳定的自动化发布检查。

### Phase 12：模板与客户价值

目标：验证饭单复用、客户偏好沉淀和专业服务价值。该阶段先进入 Backlog，待家庭工作区真实试用后再排期。

- [LES-116](https://linear.app/less-lab/issue/LES-116) 实现模板库和一键创建流程。
- [LES-117](https://linear.app/less-lab/issue/LES-117) 增强用餐对象档案与历史偏好。
- [LES-118](https://linear.app/less-lab/issue/LES-118) 支持从访客反馈直接调整饭单。
- [LES-119](https://linear.app/less-lab/issue/LES-119) 增加协作提醒与通知偏好。

## 4. 数据模型方向

Phase 9 预计新增或调整：

- `space_members`：`space_id`、`user_id`、`role`、`status`、`joined_at`。
- `space_invitations`：`space_id`、`token`、`role`、`invited_by`、`expires_at`、接受/撤销状态。
- 当前工作区选择：使用安全、可恢复的服务端会话或用户偏好，不依赖不可信的查询参数决定权限。
- 现有 `spaces.owner_user_id`：迁移期保留兼容，同时为 owner 回填成员记录；后续是否移除另行决策。
- 审计字段：Phase 10 为关键记录补充 `created_by`、`updated_by` 或独立活动表。

所有创建者侧业务查询仍必须通过当前 `space_id` 隔离。公开分享页继续通过分享 token 读取访客安全数据，不复用成员权限。

## 5. 明确不做

1.1 不包含：

- 实时光标、实时文本级共同编辑。
- 复杂企业角色和字段级权限。
- 完整库存和冰箱管理。
- 精确营养或医疗级建议。
- 支付订单和上门服务交易撮合。
- 菜谱社区或 AI 自动推荐。

## 6. 每张任务的交付规则

每张 Linear 任务完成时必须同时：

1. 更新对应代码与 migration。
2. 更新功能文档和 `docs/development/progress.md`。
3. 运行 `npm run check`、`npm run build` 和与风险匹配的 smoke。
4. 涉及家庭工作区时至少验证两个登录账号和跨空间拒绝用例。
5. 提交代码后在 Linear 留下实现、验证和文档记录，再标记 Done。
